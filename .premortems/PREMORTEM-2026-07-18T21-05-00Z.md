### Finding 1: Reply queues expired mid-batch after the batch ceiling was raised

**Category:** 8 — Load-bearing defaults (compounded by 7 — Invisible invariants)
**Severity:** high
**Confidence:** confirmed
**Location:** `config.py:217-222`
**Mitigation verified absent:** The correctness relation `REDIS_RESULT_QUEUE_TTL_SECS >
BATCH_MAX_DURATION_SECS` is asserted only in `tests/test_result_queue_ttl.py:37` and
`:44`. Searching the non-test tree for any runtime enforcement of that relation returned
**0 matches**; the only `raise ValueError` in `config.py` was line 83, guarding the
unrelated `PJE_BASE_URL`. Both sides of the inequality are independently settable via
`os.getenv` (`config.py:199` and `config.py:217-222`), so the derivation is only ever the
*default*.

#### Failure narrative

A large ingestion run needed more than an hour, so an operator raised
`BATCH_MAX_DURATION_SECS` to `86400` in the deploy environment and left the reply-queue
TTL at its previously-pinned `5400`. Nothing complained: the suite stayed green because
pytest imports `config` with the built-in defaults, where the derived TTL is comfortably
above the ceiling. Ninety minutes into the next long batch, the reply queue expired
underneath the dashboard's `blpop`. The worker kept downloading and kept RPUSHing —
each write re-created the key and re-armed the TTL — but every result written before the
expiry was gone. The dashboard drained a queue that had silently lost its history, hit
`RESULT_WAIT_TIMEOUT_SECS`, and marked the batch `failed`. The files were on disk the
whole time. The team spent the incident re-reading the *worker*, because the worker's
logs showed successful publishes for the very processos the dashboard reported as
failed — the same misleading signature as the 2026-07-18 BLPOP incident this change was
written to prevent.

**Hardening:**
1. Enforce the invariant at import in `config.py`, in the same fail-fast idiom already
   used at `config.py:83`, so a bad combination aborts the process loudly instead of
   losing data quietly. **(Applied in this branch — `config.py:224-235`.)**
2. Cover the enforcement with tests that reload `config` under overridden env, since the
   default-valued asserts structurally cannot see this class of failure.
   **(Applied — `tests/test_result_queue_ttl.py::TestTTLInvariantEnforcedAtRuntime`.)**
3. Consider the symmetric guard against `RESULT_WAIT_TIMEOUT_SECS` if that value ever
   becomes independently tunable beyond the batch ceiling.

### Finding 2: The leak returns silently if a future edit drops the EXPIRE

**Category:** 10 — Version-coupled assumptions
**Severity:** medium
**Confidence:** confirmed
**Location:** `worker.py:85-100`
**Mitigation verified absent:** `rpush_with_ttl` depends on three `redis.asyncio`
behaviours that are incidental rather than contractual: `pipeline()` returning an async
context manager, command queueing being *synchronous* (`pipe.rpush(...)` is not
awaited), and `transaction=True` yielding MULTI/EXEC. A grep of `metrics.py` for any
gauge or counter describing reply-queue key count or TTL returned **0 matches**, and
`ops/monitoring/` contains no alert rule for immortal keys. Nothing observes the
property the change exists to guarantee.

#### Failure narrative

A later redis-py major bump made pipeline command queueing awaitable. `pipe.rpush(...)`
without `await` became a no-op coroutine; `pipe.expire(...)` likewise. `execute()` still
returned cleanly, the unit tests still passed against the pipeline-shaped fake in
`tests/test_worker.py`, and the real-socket test happened to be skipped in the
environment where it was run. Reply queues quietly went back to `ttl=-1`. Nobody noticed
for weeks — the leak has no symptom until Redis memory pressure starts evicting live
keys, at which point the failure surfaces as *unrelated* batches losing results. This
repo has already been burned once by redis-py 8.0.0 silently changing a default
(`socket_timeout` `None`→`5`, see `config.py:170-190`), which is exactly why this
coupling deserves an alarm rather than trust.

**Hardening:**
1. Export a gauge for the number of `kratos:pje:results:*` keys with `ttl == -1` and
   alert on `> 0`; the invariant then fails loudly instead of accumulating.
2. Keep the real-socket assertion in `tests/test_result_queue_ttl.py:83-96` un-skippable
   in CI (the `ports:` mapping added in PR #32 already ensures it runs there — verified:
   CI reported `451 passed` with **0 skipped**, versus 449+2 locally).
